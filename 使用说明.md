# Lark-Formatter 用户手册

本文档旨在为使用者提供 Lark-Formatter 的标准操作规范。当前 `0.20 LTS` 版本是一款**毕业论文格式专项修订**桌面客户端，致力于实现毕业论文 `.docx` 文件的自动化、合规排版。

> 版本说明：`0.20 LTS` 仅面向毕业论文场景；其他场景拟在 `1.00` 版本完成开发与发布。

---

## 1. 运行环境配置与系统启动

### 1.1 依赖初始化
在项目根目录中，通过拥有相应执行权限的终端环境运行以下脚本，以部署运行所需的依赖库（基础环境要求为 Python 3.10 及以上版本）：
```powershell
.\install_env.bat
```

### 1.2 启动图形界面
依赖环境部署完毕后，执行以下指令启动主控制台（GUI）：
```powershell
.\start_app.bat
```
*(注：若需捕获底层引擎输出的运行时日志以供调试或故障排查，请通过执行 `.\start_app_debug.bat` 按调试模式启动。)*

## 2. 标准化排版工作流

Lark-Formatter 的引擎调度遵循高度结构化的流水线（Pipeline）模式。具体操作标准流程如下：

1. **载入目标文档**：在主界面的文档加载模块中，导入待处理的 `.docx` 源文件。
2. **挂载论文模板**：根据论文排版规范（如不同高校或院系要求），在左侧模板仓库中选用适配的毕业论文模板。
3. **限定作用域（可选）**：利用“分区设置”模块，通过物理页码定位、正则表达式关键字或段落索引，精确框定正文解析边界。此举可有效隔离前言、目录等无须重新格式化的敏感区域。
4. **触发引擎调度**：执行“开始排版”指令。系统将在后台依据既定逻辑，依次触发涵盖页面几何参数、标题提取、段落级样式重置、目录拓扑更新等十余项规则算法。
5. **获取工程产出**：调度任务归档后，可直接在源文档所处的资源目录下获取已完成处理的目标文件与相关审计附件。

---

## 3. 输出物与防篡改机制

为保障数据的绝对安全，排版引擎采取严格的**非破坏性隔离策略**。原始输入文件始终处于介质保护下，禁止任何覆写行为。流水线闭环后，产生的标准化文件包括：

*   **核心交付物**：在原始文件同目录下，生成更名附加 `_new.docx` 标识的最终排版文档。
*   **工程附件集**：系统将自动创建名为 `[原文件名]_排版附件` 的子目录，涵盖：
    *   **版本对照追溯稿**（`_对比稿.docx`）：利用 Word 原生修定跟踪机制，高亮标记格式演进与文本差异，以支持严密的二次人工校验。
    *   **执行与审计报告**（`_报告.json` / `_报告.md`）：结构化输出规则拦截日志、执行时序消耗、及上下文异常警告（例如非法悬挂标题、系统基准样式缺失等）。

## 4. 核心调度规则与实验室扩展

### 4.1 基础模块集
在常规工作流中，引擎会依次启用如下核心格式化修正：
*   **文档几何形态重建**：重设页边距规格与纸张走向，并依据正文起端自动解耦文档的页眉页脚域链接。
*   **拓扑解析与大纲重构**：依靠启发式解析树关联内置样式模型（支持重构 Heading 1-9 及其联动的多级编号系统）。
*   **数据结构与标号标准化**：规约数据表样式（支持标准化三线表模式）并重置冗余图表题注及占位符格式。
*   **域指针自动化重构**：格式化工程末期，系统将通过 COM 接口通知 Word 内核强制刷新全文目录（TOC）及所有的图表交叉引用缓存。

### 4.2 实验室特性 (Labs)
针对部分携带非结构化特征的草稿数据，系统内置了扩展实验性算法（**建议实施小样本验证无误后，再接入批量化生产调度**）：
*   **Markdown 语法规约化**：利用正则引擎清洗并解析正文内附带的加粗（`**`）或标题（`#`）等纯本文标记。
*   **空白与全半角规范**：统一特殊空格、Tab 与零宽字符，折叠连续空格，并可按语境转换标点/括号/引号/全角英数（不处理空段落折叠）。
*   **引用域自动溯源**：在全局扫描中自动将纯文本编号（例如 `[1]`）投射并关联至尾部参考文献的真实交叉引用域。
*   **理工类公式智能修正**：通过模式探勘识别特定的化学分子式及数理表达，自动应用正确的上下角标并予以对齐规范。

## 5. 论文模板存储架构与样式克隆

为了支持毕业论文规范的复用与迭代，架构内建了模板化配置机制。

*   **结构克隆**：用户可通过系统内“克隆模板”特性指令，输入具备标杆排版的毕业论文参考文档，系统将自动抽取其底层样式字典、页面空间设定与编号拓扑体系，并存入库中作为新的论文模板。
*   **存取路由**：所有的新增及用户态变更将会持久化留档于二进制执行文件 `Lark-Formatter.exe` 同级的 `templates` 主目录下。
*   **核心资源防破坏**：若在冷启动序列中侦测到基本排版池残片化，恢复管理器会自动从内置沙箱 (`_internal/templates`) 降级下发拷贝用以补充结构完整度，且该行为具备防覆写锁。

> 当前模板体系仍以毕业论文场景为核心，不建议将其视为面向所有文档类型的通用模板平台。

## 6. 高阶环境变量与开发者调用

针对有 CI/CD 环境接入或深层控制偏好的进阶开发者，现开放如下操作系统级环境变量以接管程序执行策略：

*   `DOCX_FORMATTER_TEMPLATES_DIR`：截获并强制重定向自定义的模板存取命名空间寻址，尤为适合分布式云端驱动的使用场景。
*   `DOCX_PIPELINE_STRICT_MODE`：向其注入布尔真值 `1` 可激活管道严格容错策略。在流转过程中若核心模块抛出致命级别的断言错误，管线将即时进入熔断状态并取消变更提交。
*   `DOCX_DISABLE_FIELD_REFRESH`：向其注入布尔真值 `1` 可强制旁路管道尾部的 Word COM 进程挂载。该行为旨在根除异步子进程陷入死锁的风险，相关目录刷新需转由人工介入。

如需自行编译封装独立可执行版本，则仅需在项目主干下触发脚本：
```powershell
.\package_release.bat
```
*(成功编译的 Windows 发布包将归档输出至 `dist/Lark-Formatter_v0.20_LTS/` 目录)*
